# Domain Dictionary

**Feature Spec**: Full specification for a new feature, resulting in a complex `tasks.md` defining the execution DAG. _Avoid_: especificação genérica, plano longo.
**Mini-DAG**: Simplified, sequential task table for hotfixes/quick patches saved in `.specs/hotfixes/`. _Avoid_: quick mode output, small spec.
**Continuous Flow**: Journey where Orchestrator generates the Feature Spec or Mini-DAG and immediately launches execution in the same session. _Avoid_: execução acoplada, jornada 1.
**Disjoint Flow**: Journey where Orchestrator executes a previously generated Feature Spec or Mini-DAG in a new, independent session. _Avoid_: execução desacoplada, jornada 2.
**Intentional Manual Execution**: Running execution scripts (like `./scripts/run_dag.sh`) manually in the user's terminal to preserve observability and trust, rather than autonomously inside a sandbox. _Avoid_: execução autônoma, ctx_execute.
**Escalation Phase**: Formal recovery protocol triggered when a worker task fails repeatedly. Orchestrator reads the failure log and plans a hotfix. _Avoid_: manual fixing, ad-hoc patching.
**Double Verification**: Two-step validation during Escalation: first running the Mini-DAG execution, then manually running `auditor.sh` against the main DAG's gate to prove the hotfix works before resuming. _Avoid_: single check, blind resume.
**Core Infrastructure**: The mandatory duo of tools (`rtk` for token-optimized CLI execution, and `context-mode` for Orchestrator out-of-band file parsing) required to run `dag-flow`. Verified by `dag init` before installation proceeds. _Avoid_: ferramentas opcionais.
**Init Idempotency**: `dag init` aborts if a Compiled Skill already exists at the target path; `dag init --force` overwrites (local edits lost); `dag update` is the only normal upgrade path. _Avoid_: silent overwrite, re-init on every version bump.
**Init Prerequisites Gate**: The `dag init` CLI check that aborts unless `context-mode` and `rtk` are available on PATH, with actionable install instructions. Infrastructure validation belongs to the CLI — not to SKILL.md instructions the Orchestrator reads. _Avoid_: soft warning, prerequisite lines in skill.
**Project Scaffold**: The complete `.specs/` topology `dag init` creates idempotently in the target project: `.specs/staging/` (chmod 755), `.specs/features/` (chmod 555), `.specs/dags/` (chmod 555). Replaces partial bootstrap from removed `setup_indexer.sh`. Runtime dirs (`.specs/runs/`, `.specs/hotfixes/`) are created on first use — not part of init scaffold. _Avoid_: dags-only bootstrap, user-created vault dirs, chmod left to the Orchestrator.
**Discovery Phase**: Optional standalone operation where the Orchestrator uses `ctx_search` to synthesize Architectural Invariants into `CONTEXT.md`. Triggered only on explicit user request. Protocol lives inline in `SKILL.md` — no separate `references/discovery.md`. _Avoid_: Map Phase, global crawling, auto-trigger on Specify.
**dag-flow Hook**: Centralized Guard+Router logic in `cli/hooks/dag-flow-guard.mjs` (npm CLI — not Compiled Skill). **Guard:** blocks writes to functional code paths (`src/`, `lib/`, `api/`, `test/`) unless `DAG_FLOW_WORKER=1`. **Router:** on block, infers Specify vs Quick Mode from the user prompt and emits a single adaptive message — no session-state detection. Quick Mode when the prompt contains a canonical trigger phrase OR (1–2 files + error/symptom/regression + no new requirements); otherwise Specify — bias toward Specify when ambiguous. Wiring to the Orchestrator Runtime is declared by **Hook Wiring Tier** in the orchestrator manifest — not hardcoded per runtime name. _Avoid_: global indexing hook, context-mode setup, session flag files, dual message paths, Antigravity-specific hook logic.
**Hook Wiring Tier**: Orchestrator manifest field declaring how `dag init` attaches the dag-flow Hook. **`pre_tool_use`:** upsert into orchestrator hook config at `HOOK_CONFIG_PATH` — physical Guard. **`boot_file`:** idempotent append of cognitive rule to `BOOT_FILE` — no physical Guard. **`none`:** skip wiring; CLI warns — no Guard. V1 built-ins: `cursor` → `pre_tool_use`; `antigravity` → `boot_file`. Custom manifests inherit tier via `extends`. _Avoid_: per-runtime ad hoc wiring, Antigravity exception, hook logic in Compiled Skill.
**Hook Config Upsert**: Idempotent merge protocol for tier `pre_tool_use`: `dag init` / `dag update` read `HOOK_CONFIG_PATH`, upsert only the entry identified by `HOOK_ENTRY_ID` at `HOOK_EVENT`, and never remove or overwrite third-party entries. JSON shape per runtime is declared in the built-in manifest adapter — not hardcoded in CLI. _Avoid_: overwriting hook config, Cursor-only merge, manual post-init wiring.
**BOOT_FILE**: Orchestrator-runtime boot file path in the manifest (e.g. `GEMINI.md`). Used by **`boot_file`** Hook Wiring Tier to persist cognitive Guard+Router rules when the runtime has no native pre-tool hook API. Not for onboarding docs. _Avoid_: getting-started compile target, worker config, indexing setup.
**Architectural Invariants**: The synthesized, high-level map of the project structure, sub-contexts, and technical stack stored physically in `CONTEXT.md` and `docs/adr/`. _Avoid_: Context Map, High-Level State.
**Living Memory**: The ongoing property of keeping Architectural Invariants synchronized with reality via T-Final Delta Updates after execution, avoiding full re-scans. _Avoid_: Map Phase (as a continuous state).
**Design Phase**: The architectural step between Specify and Tasks that defines the concrete technical interfaces, components, and patterns to be implemented. _Avoid_: pattern definition only, abstract architecture.
**DAG Validator Gate**: A mandatory shell-level interceptor script (`scripts/write_dag.sh`) that validates the DAG table schema before saving it. *Deprecated: See Bash Gates*.
**Bash Gates**: Interceptor shell scripts (`scripts/commit_spec.sh`, `scripts/commit_design.sh`, `scripts/write_dag.sh`) that run Python validation on `.specs/staging/` files and act as the sole authorized writers moving artifacts to the physical vaults. _Avoid_: soft validation, read-time gate.
**Staging Area**: The `chmod 755` directory (`.specs/staging/`) where the Orchestrator drafts markdown artifacts and PAGRL XML traces before they are intercepted and validated by Bash Gates. _Avoid_: draft folder, work dir.
**Subagent Relay**: The architectural pattern of delegating DAG generation to an isolated Subagent Planner (`enable_write_tools=false`) to physically prevent the Orchestrator from breaking OS-level locks. _Avoid_: orchestrator writing.
**Topology Separation**: The architectural division of artifacts where free-text specs and executable JSON DAGs live in separate, locked directories (`.specs/features/` and `.specs/dags/`), enabling OS-level physical gating from the open staging area. _Avoid_: unified specs folder, open directories.
**Isolated Task Logs**: The mechanism of capturing stdout/stderr of concurrent execution workers into separated `.specs/runs/[id]/logs/T[ID].log` files to preserve observability and prevent race conditions during the Escalation Phase. _Avoid_: live console streaming, interleaved outputs.
**DAG Runner Engine**: A Python-based topological executor that replaces the pure Bash sequential loop, orchestrating independent tasks concurrently while preserving sandbox boundaries. _Avoid_: Bash loop execution, sequential runner.
**Drain-running**: A fail-safe task management strategy where, upon a DAG failure (a task exhausting its attempts), running parallel tasks are allowed to complete their execution atomically to prevent file corruption, but no new tasks are dispatched. _Avoid_: fail-fast, immediate abort, SIGTERM.
**Worker Pool**: A concurrency limiter in the DAG Runner Engine (defaulting to 3) that queues independent tasks to prevent CPU/RAM exhaustion and LLM API rate limits. _Avoid_: unbounded concurrency, unlimited background jobs.
**JSON DAG**: The executable AST (Abstract Syntax Tree) format for DAGs (`dag.json`), replacing legacy Markdown tables. It natively enforces structural schema adherence and embeds cognitive rationale fields directly into the task nodes. _Avoid_: Markdown tables, unstructured tasks.
**DAG Planner PAGRL**: The XML cognitive trace (`planner.pagrl.xml` with `<PAGRL phase="DagPlanner">`) emitted by the Subagent Planner before generating the JSON DAG. Forces explicit reasoning about validation strategies before task execution. _Avoid_: merging with Orchestrator's `tasks.pagrl.xml`, unstructured reflection.
**Determinism Rule**: The architectural principle requiring the Orchestrator to map any subjective code change intent to literal mathematical coordinates (exact strings, regex, AST nodes) before delegating to workers, avoiding *Context Blindness*. _Avoid_: example-based instructions, delegated semantic interpretation, LLM text validation.
**Common Ground Flow**: The mandatory two-phase process during Specify where the Orchestrator first surfaces assumptions into `common_ground.md` and halts execution, requiring explicit human approval before generating `spec.md` and running the Bash Gate. Defeats LLM complacency. _Avoid_: zero-shot specs, syntactic XML validation for cognitive states.
**Orchestrator Runtime**: The agent environment where the user interacts and the dag-flow skill is loaded (e.g., Cursor IDE, Antigravity). Supplies Category B tool names, Category C descriptive text, install path, Hook Wiring Tier, `HOOK_CONFIG_PATH` / `HOOK_EVENT` / `HOOK_ENTRY_ID` (tier `pre_tool_use`) or `BOOT_FILE` (tier `boot_file`), and spawn-subagent partials during compilation. _Avoid_: worker runtime, execution runtime.
**Worker Runtime**: The CLI binary spawned by the DAG Runner Engine to execute each task in isolation (e.g., `agy`, `cursor agent`). Supplies Category A CLI placeholders (`CLI_COMMAND_PREFIX`, `AUDITOR_COMMAND_TEMPLATE`) and `dag-config.json` `command_template`. Permission flags differ per worker — see TDD §5.13 (`--dangerously-skip-permissions` for `agy`; `--print --trust --force --approve-mcps` for `cursor agent`). May differ from the Orchestrator Runtime; when unspecified, defaults to the same runtime as the Orchestrator. _Avoid_: orchestrator runtime, the main agent, reusing agy flags on cursor.
**Dual Manifest Compilation**: The `dag init` process that resolves two Runtime Manifest Directories when orchestrator and worker differ — orchestrator manifest for tool names and install path; worker manifest for CLI command templates embedded in Compiled Skill markdown and for `dag-config.json`. When `--worker` is omitted, a single manifest serves both roles. _Avoid_: single-manifest compile, orchestrator-only substitution.
**Runtime Manifest**: The per-runtime configuration record in the dag-flow source that maps semantic placeholder identifiers (`<<<DAG:NAME>>>`) to their runtime-specific values. _Avoid_: runtime config, hashmap.
**Source Skill**: The runtime-agnostic source files in the dag-flow development repo, containing `<<<DAG:NAME>>>` placeholders instead of runtime-specific values. _Avoid_: compiled skill, installed skill.
**Compiled Skill**: The output of `dag init` — runtime-resolved Agent Skill package installed at the orchestrator's skill install path (e.g. `.cursor/skills/dag-flow/`). Contains only operational artifacts: `SKILL.md`, `references/`, **Compiled Skill Scripts** (whitelist), and `dag-config.json`. Onboarding docs (e.g. `getting-started.md`) stay in the dev repo or npm README, not in the Compiled Skill. _Avoid_: source skill, template, task skills, compiled docs, dev-only scripts.
**Compiled Skill Scripts**: Explicit whitelist of `scripts/` files copied into the Compiled Skill by the runtime compiler — operational pipeline only (`dag_runner`, gates, auditor, validators). Excludes `migrate_*` and `test_*` dev tools. Golden-gate parity compares this set, not a full manual repo copy. _Avoid_: copy all scripts, blacklist-only filtering, migrate scripts in installed skill.
**Task Skill**: An optional skill referenced by a DAG task node (`skill` field) to give workers domain-specific guidance (e.g. `coding-guidelines`). Not part of dag-flow; the user is responsible for making Task Skills available to worker agents in their environment. _Avoid_: dag-flow skill, compiled skill, MCP skills.
**Pending DAG Guard**: The rule that `dag update` scans `.specs/dags/*.json` and fully aborts if any task status is not `"Done"` — including `Pending`, `Failed`, and absent status. Task status is authoritative on the vault file; the DAG Runner Engine updates the vault (with temporary `chmod` unlock) during execution, not only the run snapshot in `.specs/runs/`. On allowed update: recompiles Compiled Skill markdowns, preserves `dag-config.json` worker section, and re-wires dag-flow hooks idempotently from the npm package. _Avoid_: partial update, runs-only status, hooks-only-on-init, allowing update with Failed tasks.
**Runtime Manifest Directory**: A directory containing `manifest.json` (scalar values and `command_template`) and an optional `partials/` subfolder (complex multi-line markdown blocks). Custom manifests can declare `"extends": "<built-in-name>"` to inherit all values from a built-in runtime and override only what differs. Created via `dag manifest scaffold`, which requires `--extends`. _Avoid_: single manifest file, flat config, scaffold without base runtime.
**Manifest Inheritance**: The mechanism by which a custom Runtime Manifest Directory declares `"extends": "<built-in>"` to inherit all scalar values and partials from a built-in runtime, overriding only the fields that differ. Eliminates the need to understand internal block structure when adding a new similar runtime. _Avoid_: full manifest from scratch.
**Built-in Runtime**: A first-party Runtime Manifest Directory shipped in `cli/manifests/`. **V1 ships two:** `antigravity` (parity / golden gate) and `cursor` (first multi-runtime target after infra). `claude` is deferred post-V1 — extensible later via `extends` or custom manifest. _Avoid_: gemini-cli, shipping claude in V1 scope.
**Auditor Gate Contract**: The invariant rule that `auditor.py` expects worker CLI subprocess stdout to contain `PASS` (and exit code 0) when a task's `done_when_gate` succeeds. Runtime-specific output quirks are the user's problem — resolved via **Worker Runtime Validation Spike** or an optional `auditor_wrapper` script in the worker manifest, not by making `auditor.py` runtime-aware. Worker permission flags per runtime: **ADR-0017**. _Avoid_: per-runtime auditor, stdout-agnostic validation, `--dangerously-skip-permissions` on cursor.
**Worker Runtime Validation Spike**: Minimal manual experiment run before publishing an unvalidated Worker Runtime manifest: invoke the worker CLI with a trivial prompt (`Respond EXACTLY: PASS`), record exit code and stdout, document pass/fail in the TDD. Gates only that runtime's manifest publication — not shared infrastructure. On failure: ship `auditor_wrapper` in the worker manifest. _Avoid_: full e2e benchmark, global implementation blocker, orchestrator runtime testing.
