# 5. OS-Level Gating and Topology Separation

Date: 2026-06-12

## Status

Accepted

## Context

Following the `e2e-0.1.0` benchmark failure, it was observed that pure prose and prompt-level constraints (like the Counted PAGRL mechanism introduced in ADR-0004) were insufficient to completely prevent LLM agents from improvising DAG tables or writing them to incorrect locations. When the model's training bias to answer directly overcomes the prompt instructions, it outputs the tasks arbitrarily, which breaks the automated pipeline.

We need a physical, deterministic mechanism to ensure that the DAGs generated by the Orchestrator strictly adhere to the required 9-column schema, include the necessary terminal verification gates (`T-Final`), and are written to the correct location for the runner to pick up, without relying purely on cognitive discipline from the LLM.

## Decision

We are implementing Solution (c) originally discussed in ADR-0004: **deterministic gates implemented in shell scripts that intercept artifact writes.**

To support this:
1. **Topology Separation**: We separate free-text specifications (`.specs/features/`) from executable DAGs. DAGs will now be stored in a dedicated, physically locked directory: `.specs/dags/`.
2. **Physical Lock**: The `.specs/dags/` directory is created with `555` (`r-xr-xr-x`) permissions by default via `hooks/setup_indexer.sh`, making it impossible for the LLM to write to it using standard file-editing tools.
3. **DAG Validator Gate**: We introduce `scripts/write_dag.sh`, a shell-level interceptor. This script is the sole authorized writer to the vault. It temporarily elevates directory permissions (`chmod 755`), validates the incoming markdown table for structural correctness (9 columns, presence of `T-Final`, and proper override flags), writes the file, and restores the lock (`chmod 555`) using a crash-safe `trap`.

## Consequences

- **Positive**: Complete eradication of malformed DAGs being saved to disk. If the LLM improvises the format, the gate script rejects it, forcing a backprop reflex or failure.
- **Positive**: Strict decoupling of readable specs and executable workflows at the file-system level.
- **Negative**: Adds a layer of indirection (a third script) to the dag-flow pipeline, marginally increasing complexity.
- **Negative**: Modifies the paths in the orchestration memory and benchmarks, requiring a migration of existing `evals/`.

This deterministic mechanism guarantees that the `run_dag.sh` engine will always receive structurally sound execution plans, shifting the enforcement of phase discipline from prompt engineering to operating system permissions.
