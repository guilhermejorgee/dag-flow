# Benchmarks & The End of Ossification

## The Fallacy of Context Windows

As Large Language Models (LLMs) expand their context windows, developers increasingly rely on conversational agents to build entire features in a single prompt. While this works for trivial scripts, it fails catastrophically for complex architectures due to **Monolithic Dumping**, **LLM Overconfidence**, and **Few-Shot Ossification**.

To prove that `dag-flow` systematically eliminates these failure modes, we conducted two tiers of rigorous evaluation: a quantitative End-to-End (E2E) Exhaustive Benchmark, and a qualitative Architectural Showdown.

---

To prove the robustness of `dag-flow` across edge cases, we developed a comprehensive E2E Harness (`e2e-v0.1.0`) that simulates real-world developer interactions and strictly enforces architecture constraints.

**The Test Harness Engine:**
1. **Pyte Socratic Emulator:** Simulates the user interacting with the CLI's TUI via Bubbletea, actively answering the Orchestrator's questions without manual intervention.
2. **Double Vaulting Compliance:** The harness explicitly enforces file permission locks (`chmod 555` on `.specs/features/`) preventing the AI from cheating. The agent is forced to use the Staging Area and Bash Gates.
3. **Automated LLM Grading:** Assertions check file contents via globs, regex, exact matching, and dispatch a deterministic LLM judge to evaluate the qualitative aspects of the generated specifications.

**The Results:**

## 🕵️ Spotlight: Beating Hallucination in Real-Time

A major cause of LLM failure is **blind coding**—writing code to satisfy a prompt without checking if the architectural dependencies actually exist. During Scenario 5 (`ambiguous-spec`), the agent was instructed to implement an in-app notification system.

A traditional agent would immediately hallucinate a database table and start writing SQL or Mongoose schemas. But because the `dag-flow` Orchestrator is governed by the PAGRL gate (which forces reasoning before execution), it stopped the pipeline entirely. 

**Here is the exact transcript extracted from the benchmark execution log:**

> **Orchestrator Thought Process:**
> `Recipient logic noted. Dictionary updated. Cannot start coding. App has no database configured. How should we persist these notifications: in-memory, SQLite, PostgreSQL, or something else?`
>
> **Socratic Output to User:**
> `[Cannot start coding] [App has no database configured]. [How should we persist these notifications?]`
> 
> **User / Emulator Reply:**
> `Let's keep this simple and start with an in-memory solution for now. We can store the notifications in a per-user array.`

Instead of blindly hallucinating infrastructure, the Orchestrator recognized the missing architectural constraint, suspended the pipeline, and interrogated the user. This is the difference between an AI that types fast and an AI that acts like a Senior Engineer.

> [!NOTE]
> **Why an in-memory solution?** 
> The reply above was generated by the Socratic Emulator, which was explicitly configured in the test harness (`scenario.json`) to request an in-memory solution. This was designed to test if the Orchestrator would respect a user's architectural constraint rather than falling back on standard LLM training data (which almost always defaults to hallucinating an SQL or MongoDB setup).

## ⚔️ Part 2: The Qualitative RBAC API Showdown

To understand *why* the baseline fails and dag-flow succeeds, we asked two isolated agents to build a Role-Based Access Control (RBAC) system in Node.js/Express.

**The Strict Requirements:**
- `POST /login` (returns JWT).
- Role hierarchy: Admin, Editor, Viewer.
- Middlewares to decode JWT and guard routes.
- Native unit tests covering authorization failures and successes.

### 🚜 The Baseline Agent (Conversational LLM)
We provided the exact requirements to a top-tier LLM Agent without orchestration skills.

**The Result:** The LLM successfully passed the prompt constraints, but produced an architectural nightmare. It dumped the database mock, the authentication middleware, the role-guarding logic, and all Express routes into a **single monolithic `app.js` file** of over 150 lines. The tests were highly biased because the same context window that wrote the spaghetti code also wrote the assertions.

### ⚙️ `dag-flow` (Orchestrated SDD)
We provided the exact same requirements to an Agent equipped with the `dag-flow` skill, forcing it to act as the Executive Orchestrator.

**The Result:**
1. **Domain Isolation:** The Orchestrator mapped the requirements and generated a strict `CONTEXT.md` and `spec.md`.
2. **Architectural Design:** It generated a `design.md` explicitly deciding to split the logic into layers.
3. **Execution DAG:** It generated an executable `tasks.md` separating the DB, the middlewares, the routes, and the tests into atomic tasks.
4. **Decentralized Workers:** The `run_dag.sh` script dispatched stateless CLI workers to write each file in complete isolation.
5. **Independent Auditing:** The tests (`api.test.js`) were written by a worker that had *only* the finished `app.js` in its context, structurally eliminating test bias.
